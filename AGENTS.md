# Agent 工作指南

## 核心原则

**workflow enforcement**: research → plan → implement

所有有意义的修改必须遵循此顺序，在plan前先写research，且必须在用户批准 plan 后才能编写代码。

---

## 文件命名规范

- **Research 文件**：`<YYYY-MM-DD>-<改动名称>-research.md`
- **Plan 文件**：`<YYYY-MM-DD>-<改动名称>-plan.md`

示例：`2026-06-02-auth-migration-research.md`

---

## 工作流程

### 1. Research（研究阶段）

深度阅读代码库相关部分，输出 Research 文件，包含：
- 核心文件路径
- 现有逻辑潜在隐患
- 技术链路分析

### 2. Plan（规划阶段）

输出 Plan 文件，包含：
- 架构设计与逻辑流
- 受影响文件列表
- 代码片段示例
- 破坏性变更评估
- 注意事项与折衷方案

**注释循环**：Plan → 用户审查添加内联注释 → 更新计划 → 循环直到用户满意  
**保护指令**：`don't implement yet` — 收到此指令时禁止编写代码，仅更新计划  
**Todo List**：在 Plan 文件末尾添加带复选框的任务清单

### 3. Implement（实施阶段）

**启动指令**：`implement it all`

**执行标准**：
- 完整执行计划所有内容
- 完成任务后在 Todo List 打勾标记
- 禁止使用占位符（`// TODO`、`// ...`）
- 严格错误处理，禁止隐式吞掉 Error
- 禁止使用 `any` 或 `unknown` 类型
- 保持代码简洁，不添加冗余注释
- 持续运行 typecheck

**反馈响应**：对简短反馈立即调整；收到回滚指令时停止原计划，专注新需求  
**用户决策**：用户拥有最终决策权，可能选择性采纳、缩减范围、保护现有接口或覆盖技术选择

---

## 工作模式

单个长会话完成：深度阅读 → 编写计划 → 多轮注释更新 → 全量实施并保持类型正确

---

## 行为准则

1. **编码前思考**：明确假设，不隐藏困惑，权衡取舍
2. **简单优先**：用最少代码解决问题，不做投机性抽象
3. **精确修改**：只改动必须改动的部分，匹配现有风格
4. **目标驱动**：定义可验证的成功标准，循环直到通过

---

**准则生效标志**：diff 中不必要的更改更少，澄清问题在实现前提出
